안녕하세요! 지난주 AWS에서 Kiro라는 새로운 AI IDE를 출시했다는 소식을 듣고, 주말 동안 과연 기존 AI 코딩 도구들과 얼마나 다를지 직접 테스트해보고 싶다는 생각이 들었습니다. 그래서 간단한 사이드 프로젝트로 Next.js 기반의 기술 블로그를 만들어 GitHub Pages에 배포해 보기로 했습니다. 목표는 단 하나, "**제가 직접 코드를 거의 작성하지 않고, 오직 Kiro와 함께 블로그를 완성하는 것**"이었습니다.

이 글에서는 제가 Kiro와 함께 주말 동안 겪었던 개발 경험과 그 과정에서 느낀 점들을 공유해 보려고 합니다.

***

## 명세서 기반의 AI 오토파일럿: AWS Kiro

Kiro는 기존의 코드 자동 완성 도구나 챗봇 형태의 AI 어시스턴트와는 조금 다른 접근 방식을 취합니다. 단순히 코드 조각을 생성해 주는 것을 넘어, 프로젝트의 전체적인 **설계와 명세를 기반으로 스스로 개발 계획을 수립하고 코드를 작성**해 나가는 'AI 소프트웨어 개발자'에 더 가깝습니다.

저는 Kiro의 개발 철학을 온전히 경험해보고 싶어서 별도의 CLI 도구나 다른 도구를 연결하지 않고 오직 Kiro만으로 프로젝트를 진행했습니다.

### 1단계: 목표 설정 및 기술 스택 정의

프로젝트를 시작하면 Kiro는 가장 먼저 프로젝트의 목적, 기술 스택, 개발 및 배포 환경 등 전반적인 가이드라인을 설정하는 과정을 거칩니다. 마치 새로운 팀원에게 프로젝트를 온보딩하는 과정과 비슷해서 무척 현실적이라고 느꼈습니다.

저는 다음과 같이 우리가 사용할 기술 스택을 정의해주었습니다.

* **Frontend Framework**: Next.js 15+
* **Styling**: Tailwind CSS
* **Component Library**: shadcn/ui
* **Language**: TypeScript

<center>Kiro의 'Agent Steering' 기능을 통해 프로젝트의 기술 가이드라인을 설정하는 모습</center>

***

### 2단계: 명세 기반 개발 (Spec-Driven Development)

가이드라인 설정이 끝나면 개발 명세를 작성하는 단계로 넘어갑니다. Kiro는 이 과정을 'Spec 기반 명세'라고 부르는데, 추상적인 요구사항을 구체적인 개발 요건으로 만들어주는 점이 무척 인상 깊었습니다. 예를 들어, "팀원들이 글을 쓰고 공유할 수 있으면 좋겠다"와 같은 모호한 요구사항을 다음과 같은 BDD(행동 주도 개발) 스타일의 명세로 구체화해줍니다.

> **Requirement 1: 블로그 포스트 작성 및 발행**
>
> **User Story:** 팀원으로서, 나는 기술 지식을 팀 및 커뮤니티와 공유하기 위해 블로그 포스트를 작성하고 발행하고 싶다.
>
> **Acceptance Criteria:**
> 1.  WHEN 관리자 인터페이스에 접근하면 THEN 시스템은 블로그 포스트 작성을 위한 리치 텍스트 에디터를 제공해야 한다.
> 2.  WHEN 포스트를 작성할 때 THEN 시스템은 제목, 내용, 태그, 작성자 정보 추가를 허용해야 한다.
> 3.  ...

이렇게 정의된 명세는 이후 모든 개발 과정의 기준점이 됩니다.

위의 예시에서도 나타나듯이, Kiro에서 Spec-Driven-Development라고 정의된 내용은 우리에게 익숙한 Behavior-Driven-Development 방식과 유사한 형태로 정의되고 있어, Kiro와 함께 기획하지 않은 프로젝트의 구현을 맡길 때에도 기존 기획문서를 간단하게 주입시킬 수 있어 보였습니다.

***

### 3단계: 설계 및 구현 계획 수립

![자동으로 생성된 아키텍처](/images/blogs/aws-kiro-github-pages/mermaid-diagram.svg)

명세 작성이 완료되면, Kiro는 이를 바탕으로 전체 시스템 아키텍처, 컴포넌트 설계, 데이터 모델, 테스트 방안 등을 포함한 상세한 **설계 문서(Design Document)**와 **구현 계획(Implementation Plan)**을 자동으로 생성합니다.

![자동으로 생성된 구현 계획](/images/blogs/aws-kiro-github-pages/Screenshot%202025-07-19%20at%2010.54.52%E2%80%AFAM.png)
<center>정의된 명세를 바탕으로 Kiro가 생성한 단계별 구현 계획</center>

각 구현 Task는 이전에 정의한 요구사항(Requirements)과 명확하게 연결되어 있습니다. 덕분에 AI가 생성한 코드임에도 불구하고 전체 코드베이스의 일관성이 매우 높게 유지되었습니다. 예를 들어, 기존의 AI 코딩 방식에서는 같은 기능을 하는 컴포넌트라도 AI의 상태에 따라 코딩 스타일이 달라지는 경우가 많았는데, Kiro는 모든 작업을 명세와 설계에 기반하여 수행하므로 이런 문제가 거의 발생하지 않았습니다.

***

### 4단계: AI 에이전트의 자동 개발

이제부터는 Kiro 에이전트가 직접 코딩을 시작합니다. 저는 그저 Kiro가 생성된 Task를 하나씩 처리하며 코드를 작성하고, 프로젝트 구조를 만들어나가는 과정을 지켜보기만 하면 됐습니다.

![Kiro 에이전트가 Task를 실행하는 모습](/images/blogs/aws-kiro-github-pages/Jul-19-2025%2011-29-20.gif)
<center>Kiro가 터미널 명령을 실행하고 파일을 생성하며 프로젝트를 구성하는 모습</center>

Kiro는 현재 Claude 3.7 Sonnet과 4o Sonnet 모델을 사용하는데, 아직 프리뷰 단계라 그런지 응답 속도가 약간 느릴 때도 있었습니다. 북미 시간으로 낮이 되면 사용자가 몰려 가끔 요청 제한에 걸리기도 했습니다. 하지만 오히려 AI가 코딩하는 동안 저는 Kiro가 제안한 새로운 기술이나 라이브러리를 공부할 여유가 생겨서 나쁘지만은 않았습니다.

![Task 진행 상황 업데이트](/images/blogs/aws-kiro-github-pages/Screenshot%202025-07-19%20at%2011.30.56%E2%80%AFAM.png)
<center>하나의 Task가 완료되면, Kiro는 스스로 상태를 업데이트하고 다음 Task를 진행합니다.</center>

### 결과물: 주말 만에 완성된 기술 블로그

그렇게 이틀이라는 시간 동안, 저는 거의 코딩에 손을 대지 않고도 꽤 많은 기능이 구현된 블로그를 완성할 수 있었습니다. 실제 작업 시간은 훨씬 짧았고, 대부분은 Kiro가 작업하는 것을 지켜보며 커피를 마시는 시간이었습니다.

완성된 기능은 다음과 같습니다.
* Next.js 15 기반의 정적 사이트 생성(SSG)
* GitHub Pages 자동 배포 설정
* 한국어/영어 다국어 지원 (i18n)
* 다크 모드 지원 및 반응형 디자인
* MDX 기반 포스트 작성 및 코드 구문 강조
* 동적 OG (Open Graph) 이미지 생성
* 관리자 페이지를 통한 포스트 생성 및 관리

## Kiro 사용 후기 및 느낀 점

Kiro와 함께한 주말 프로젝트는 정말 신선한 경험이었습니다. 기존 AI 코딩 도구들이 '운전 보조 시스템' 같았다면, Kiro는 **목적지만 알려주면 알아서 운전하는 자율주행 시스템**에 가까웠습니다.

물론 아직 프리뷰 단계라 응답이 느리거나 가끔 막히는 등 답답한 부분도 있었습니다. 하지만 명확한 설계와 명세를 기반으로 전체 프로젝트의 일관성을 유지하며 개발을 진행하는 방식은 정말 강력했습니다. 개발자는 코드 작성이라는 행위에서 벗어나, **더 높은 수준의 아키텍처 설계와 요구사항 정의에 집중**할 수 있게 됩니다.

개발 도중 요구사항이 변경되었을 때도, 저는 코드 대신 **명세(Requirement) 문서를 수정**했습니다. 그러자 Kiro는 연결된 Task들과 관련 코드들을 새로운 명세에 맞게 스스로 수정해주었습니다. 이 경험은 AI와 협업하는 방식의 패러다임이 바뀌고 있음을 실감하게 했습니다.

### 남은 과제들

아직 일일 사용량 제한 때문에 계획했던 모든 기능을 구현하지는 못했습니다. Kiro가 생성한 구현 계획에는 다음과 같은 작업들이 남아있습니다.

* 에러 처리 및 로딩 상태 구현
* 개인정보 보호 정책 및 GA 동의 기능 추가
* 단위/통합/E2E 테스트 스위트 작성
* 성능 및 번들 사이즈 최적화
* 프로덕션 배포 및 모니터링 설정

이 남은 작업들도 제가 직접 코드를 수정하기보다는, Kiro가 어떻게 해결해 나가는지 지켜보면서 프로젝트를 마무리해 볼 계획입니다.

## 마무리하며

AWS Kiro와 함께한 이번 사이드 프로젝트는 'AI 기반 소프트웨어 개발'의 미래를 엿볼 수 있는 흥미로운 경험이었습니다. 비록 아직 완벽하진 않지만, 개발의 패러다임을 '코딩'에서 '설계와 명세 관리'로 전환시킬 수 있는 잠재력을 충분히 보여주었다고 생각합니다.

앞으로 Kiro와 같은 도구들이 어떻게 발전하고 우리의 개발 문화를 바꾸어 나갈지 무척 기대됩니다. 여러분도 기회가 된다면 꼭 한번 경험해 보시길 바랍니다.
